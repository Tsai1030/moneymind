# 記帳 PWA 計畫書

> 專案代號：**MoneyMind**（暫定）
> 形式：Progressive Web App（離線可用、可安裝到桌面）
> 核心理念：**輸入越快越好、痛點看得最清楚**

---

## 一、為什麼選 PWA

| 項目 | PWA 優勢 | 需要妥協的地方 |
|---|---|---|
| 跨平台 | 一份程式碼跑 iOS / Android / Desktop | iOS 對 Web Push 支援較晚（iOS 16.4+ 才有，需先「加入主畫面」） |
| 安裝成本 | 不用上架商店，網址即裝 | 需自備網域與 HTTPS |
| 開發速度 | Web 生態系成熟、迭代快 | 不能用 iOS 真正的桌面 Widget（用「捷徑 + 通知卡片」取代） |
| 更新 | Service Worker 自動更新 | 第一次載入要顧 Cache 策略 |
| 離線 | IndexedDB + SW 完整支援 | 雲端同步要自己處理衝突 |

**結論：** 先用 PWA 上線（MVP 2 個月內可驗證），未來若有需求再用 Capacitor 包成原生 App 上架。

---

## 二、技術棧建議

```
前端框架    : React + Vite + TypeScript
UI          : Tailwind CSS + shadcn/ui
狀態管理    : Zustand（輕量、學習成本低）
本地資料庫  : Dexie.js (IndexedDB 封裝)
圖表        : Recharts 或 ECharts
PWA         : Vite Plugin PWA (Workbox)
雲端同步    : Supabase (Auth + Postgres + Realtime) 或 Firebase
OCR         : Google Cloud Vision API（雲端，準確度高）/ Tesseract.js（離線備案）
語音輸入    : Web Speech API（瀏覽器內建，免費）
AI 摘要     : Claude API（產生「痛點發現」小卡）
推播        : Web Push + Service Worker
部署        : Vercel / Cloudflare Pages
```

**關鍵抉擇：**
- **離線優先**：所有寫入先進 IndexedDB，背景同步到雲端。網路斷了還能記帳。
- **本地優先 + 雲端備援**：資料的真實來源是裝置本身，雲端只是備份與多裝置同步。

## 三、功能優先級（MVP → Phase 2 → Phase 3）

### 🟢 MVP（第 1–10 週，必做才能上線）

| # | 功能 | 驗收標準 |
|---|---|---|
| M1 | 一鍵秒記輸入流程 | 從開啟到完成記帳 ≤ 3 秒、≤ 3 次點擊 |
| M2 | 帳目 CRUD + 分類管理 | 可新增、編輯、刪除、自訂分類圖示與顏色 |
| M3 | 首頁 Dashboard | 顯示「本月剩餘預算」「今日支出」「最近 5 筆」 |
| M4 | 統計頁基礎圖表 | 圓餅圖（分類佔比）+ 折線圖（每日趨勢） |
| M5 | 多帳戶切換 | 現金 / 信用卡 / 電子支付，獨立餘額 |
| M6 | 月總預算 + 進度條 | 超過 70%/90% 顯示警示色 |
| M7 | PWA 安裝 + 離線運作 | 離線可記帳、可瀏覽歷史 |
| M8 | 本地資料匯出 / 匯入（JSON） | 換手機不丟資料的保底方案 |
| **M9** | **雲端帳號 + 多裝置同步** | **Supabase Auth (Email/Google)、Last-Write-Wins 同步** |

> 為什麼把雲端拉進 MVP：你要自己用，至少手機 + 桌面瀏覽器都會開。沒同步就會「在哪台記在哪台才看得到」，那種挫折感比慢一週上線更傷。

### 🟡 Phase 2（第 11–16 週，留存關鍵）

| # | 功能 | 備註 |
|---|---|---|
| P1 | 定期支出自動化 | 房租、訂閱、保費，到日自動生成 |
| P2 | 分類預算 + 爆表警示 | 針對「外送」「娛樂」單獨設限 |
| P3 | 信用卡帳期管理 | 結帳日 / 繳費日 / 當期應繳金額 |
| P4 | 「痛點發現」AI 小卡 | 每週生成 1–3 句白話文洞察 |
| P5 | Web Push 提醒 | 預算爆表、信用卡繳費日 |

### 🔵 Phase 3（第 15 週後，差異化）

| # | 功能 | 備註 |
|---|---|---|
| F1 | 語音記帳 | Web Speech API + 規則式解析（金額/分類/品項） |
| F2 | 拍照 OCR 記發票 | Cloud Vision API，準確度比 Tesseract 高很多 |
| F3 | 共同記帳 / 分帳 | 邀請連結 + 自動結算「誰該給誰多少」 |
| F4 | 銀行 CSV 匯入 | 通用對帳工具，先支援台灣主流 3 家 |
| F5 | 載具發票同步 | 串接財政部 API（要研究授權流程） |
| F6 | 跨年度同期比較 | 「今年 5 月 vs 去年 5 月」 |

## 四、UI / UX 設計原則

### 視覺風格
- **高對比、低雜訊**：背景白/深色，文字黑/白，資料用色塊強調
- **數字是主角**：金額用大字級（48–72px），其餘資訊收進次要層
- **語意化色彩**：
  - 支出：深灰（#1F2937）/ 警示紅（#EF4444）
  - 收入：綠（#10B981）
  - 預算進度：綠 → 黃（70%）→ 紅（90%）

### 互動關鍵
- 開 App → 直接看到 Dashboard，**不要 Splash Screen**
- 按 `+` → **數字鍵盤已預設展開**，金額欄已 focus

- 長按帳目 = 編輯，左滑 = 刪除

## 五、資料模型（簡化版）

```ts
// 帳戶
interface Account {
  id: string
  name: string                // "玉山信用卡"
  type: 'cash' | 'bank' | 'credit' | 'epay'
  balance: number             // 動態計算
  currency: 'TWD'
  // 信用卡專屬
  billingDay?: number         // 結帳日
  paymentDay?: number         // 繳費日
}

// 分類
interface Category {
  id: string
  name: string                // "餐飲"
  icon: string                // emoji 或 icon name
  color: string
  parentId?: string           // 支援二層
  budget?: number             // 分類預算
}

// 帳目
interface Transaction {
  id: string
  amount: number              // 支出為負、收入為正
  categoryId: string
  accountId: string
  note?: string
  date: string                // ISO
  tags?: string[]
  // 來源追蹤
  source: 'manual' | 'voice' | 'ocr' | 'recurring' | 'csv'
  // 同步用
  createdAt: number
  updatedAt: number
  deletedAt?: number          // 軟刪除（同步衝突用）
}

// 定期支出
interface RecurringRule {
  id: string
  template: Omit<Transaction, 'id' | 'date'>
  frequency: 'monthly' | 'weekly' | 'yearly'
  dayOfMonth?: number
  nextRunAt: string
  endDate?: string
}

// 預算
interface Budget {
  month: string               // "2026-05"
  totalAmount: number
  perCategory: Record<string, number>
}
```

**同步策略：**
- 每筆都帶 `updatedAt` 時間戳
- 衝突解決：**Last-Write-Wins**（後寫贏），刪除走軟刪除避免「殭屍復活」
- 同步單位是 transaction，不要整批覆蓋

## 七、風險與對策

| 風險 | 對策 |
|---|---|
| iOS PWA 限制（沒有真正的桌面 Widget） | 改用「快速捷徑 + 鎖定畫面通知」傳達剩餘預算 |
| OCR 成本失控 | 設每月配額、優先用本地 Tesseract，準確度不夠才回落到 Cloud Vision |
| 資料同步衝突 | 軟刪除 + Last-Write-Wins，並提供「衝突歷史」可還原 |
| 用戶資料隱私 | E2E 加密選項（Phase 3）；MVP 階段至少做 HTTPS + Row Level Security |
| 自己半途而廢 | MVP 8 週砍到最小，先讓自己天天用 |

## 八、成功指標

**MVP 階段（給自己）：**
- 連續使用 ≥ 21 天（養成習慣門檻）
- 平均單筆記帳時間 ≤ 5 秒
- 漏記率 < 10%（事後對信用卡帳單）

**Phase 2 階段（給朋友 3–5 人）：**
- 7 日留存 ≥ 60%
- 每週至少有 1 次主動打開分析頁
- 至少 1 個人說：「我終於知道錢都花到哪了」

## 九、已敲定的決策

1. ✅ **設計優先**：第 1 週先在 Figma 把三個主畫面畫到能點擊，再開工。教學見 [FIGMA_GUIDE.md](FIGMA_GUIDE.md)。
2. ✅ **雲端同步進 MVP**：用 Supabase（免費額度足夠個人用）。第 9–10 週做。
3. ✅ **暫不做 Landing Page**：先自己用穩，要推廣再說。
4. ✅ **預設 8 個主分類**：
   - 🍱 餐飲、🚇 交通、🎬 娛樂、🛒 生活
   - 🏠 居家、💊 健康、📚 學習、📦 其他
5. ✅ **記帳方向採「直覺式」**：預設是支出，按金額即輸入支出；要記收入需顯式切換（因為支出記得遠比收入多）。
6. ✅ **顏色主題**：先做 Light 模式，Dark 模式 Phase 2 再加（一個人開發要省力氣）。

---

> 寫到這裡，回答你最後那題：**最容易讓人放棄記帳的，幾乎都是「分類太麻煩 + 看了沒感覺」**。
> 所以 MVP 的勝負手是兩件事：**3 秒輸入** 與 **首頁那個剩餘預算大數字**。其他功能都是錦上添花。
